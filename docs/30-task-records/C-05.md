## Scope

Formalizuje datovou vrstvu mezi C-04 sendability gate a budoucim senderem. Odděluje čtyři fáze: "lead je sendable" / "queue item čeká" / "sender posílá" / "provider potvrdil". Definuje `_asw_outbound_queue` sheet schema, queue status enum, povolené/zakázané přechody, send payload kontrakt v1.0, immediate vs scheduled pravidla, failure design a cross-ref na CS2/CS3/C-04.

Scope je **SPEC-only** — žádný runtime worker, sender, ESP provider, mailbox sync, cron ani frontend se neimplementuje. Žádné nové sloupce se v tomto tasku nezapisuji do `apps-script/Config.gs` ani do LEADS. Všechny nové sloupce jsou označené PROPOSED FOR C-05 a budou materializovány až implementačním taskem.

Task dodává:
- `_asw_outbound_queue` schema (32 polí — 15 povinných per zadání + 17 auditability/integrity rozšíření se zdůvodněním)

- Deterministický pseudocode pro queue create / worker claim / cancel / fail

**CS1/CS3 kompatibilita:**
- C-05 queue je **ortogonální** datová vrstva; nezavádí žádný nový canonical lifecycle state. T17 (`OUTREACH_READY → EMAIL_QUEUED`) mapuje na queue row insert; T18 (`EMAIL_QUEUED → EMAIL_SENT`) mapuje na queue.send_status=SENT.
- Queue status `SENDING`/`FAILED` **není** CS1 canonical state — žije pouze v queue.
- CS3 S12 `process_email_queue` pravidlo `max_attempts=1` + okamžitý dead-letter je respektováno: retry nad stejnou row je zakázán. Retry = nový queue row s jiným `idempotency_key` (jinak duplicate blocked producer-side).
- `idempotency_key` reuses CS3 section 4 S12 pattern `send:{lead_id}:{SHA256(email + subject + body)}`.

**C-04 kompatibilita:**
- Queue row smí vzniknout **pouze** pokud C-04 vrátí `AUTO_SEND_ALLOWED`. Snapshot outcome se freezne v `created_from_sendability_outcome` — audit invariant proti pozdější změně gate semantiky.
- `MANUAL_REVIEW_REQUIRED` jde do C-09 exception queue (jiná struktura, ne `_asw_outbound_queue`).
- `SEND_BLOCKED` queue row nevytváří.

## Known Limits

- SPEC neimplementuje worker / queue runtime — pouze definuje kontrakt. Implementace je budoucí task.
- PROPOSED sloupce nejsou v rámci C-05 přidány ani do LEADS, ani do `_asw_outbound_queue` sheetu — implementační task rozhodne o sheet creation a o zápisu C-04 PROPOSED sloupců (`sendability_outcome`, `sendability_primary_reason`, `sendability_evaluated_at`) do LEADS.
- C-05 nepočítá s multi-recipient scenarem per queue row (per-contact uvnitř jednoho leadu) — SPEC pracuje s "one queue row = one recipient" modelem. Future: C-07 může zavést `send_channel=EMAIL_BCC` nebo N queue rows pro thread follow-up.
- Send channel enum má zatím jedinou hodnotu `EMAIL`. Rezerva pro `SMS`/`LINKEDIN` je v schema, ne v sample ani v payload kontraktu v1.0.
- Rate limiting, quiet hours, daily caps nejsou součástí C-05 — jsou handoff do C-08. C-05 pouze drží `scheduled_at` + `priority` PROPOSED pole, aby C-08 implementace nevyžadovala schema migration.
- Suppression list (manual GDPR blacklist) je handoff do C-09. C-05 queue row pro suppressed leada nikdy nevznikne (C-04 gate ho odmítne).
- Follow-up cadence (druhý, třetí email v thread) je handoff do C-07. C-05 queue je N-queue-rows-per-lead kompatibilní (jiný subject/body → jiný `idempotency_key`).
- Rescheduling `scheduled_at` je explicitně zakázán — cancel + nový queue row je kanonická cesta. SPEC explicitně vylučuje in-place update `scheduled_at`.
- Retry nad FAILED queue row je zakázán (CS3 S12 max_attempts=1). Re-drive znamená operator intent: nový queue row. Pokud se subject/body nezměnil, producer-side duplicate guard (sekce 5) zablokuje insert.
- C-05 nedefinuje tělo `_asw_dead_letters` schema — to je CS3 section 6.3. C-05 drží pouze `dead_letter_id` cross-ref.
- Queue row `send_status` **nežije** v LEADS. LEADS drží CS1 `lifecycle_state` (future) nebo existující `email_sync_status` / `outreach_stage`. Queue je ortogonální datová vrstva. Implementační task musí řešit, jak (a jestli) zrcadlit queue.send_status do CS1 transitions.
