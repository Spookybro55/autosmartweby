## Scope

Definice kanonickeho datoveho kontraktu pro jeden scraping job. 1 job = 1 query na 1 portalu v 1 meste/segmentu. Kontrakt obsahuje 12 poli, vsechna required (key musi byt explicitne pritomen; nullable pole maji hodnotu null). Lifecycle envelope (created/running/completed/failed) a deterministicky `source_job_id` odvozeny z (portal, segment, city, district, max_results, creation second) pres SHA-256 hash10. `error_message` zachycuje chybovy detail pri stavu failed. Zadne nested objekty. Zaklad pro A-02 staging layer a A-04 scraper runtime.

## Contracts Changed

Novy kontrakt: **Scraping Job Input v1.0**.

- Vsech 12 poli je required (key musi byt explicitne pritomen v kazdem payloadu).
- Nullable pole: `district` (null = cele mesto), `job_started_at`, `job_completed_at` (null az do prechodu stavu), `error_message` (null pro ne-failed joby).
- Design decision: vsechna pole jsou required + nullable misto optional. Duvod: konzistence serializace — kazdy payload ma stejnou strukturu, zadne chybejici klice.
- Enum `portal`: `firmy.cz`, `zivefirmy.cz` (pilotni sada).
- Enum `job_status`: `created`, `running`, `completed`, `failed` se status-dependent required lifecycle timestamps vynucenymi pres `allOf`.
- `source_job_id` format: `{portal_slug}-{yyyymmddThhmmssZ}-{hash10}`, deterministicky, SHA-256 hash nad normalizovanymi scope poli.
- `max_results` rozsah 1..500.
- Schema evolution: nove portaly nebo zmena required poli vyzaduji bump `schema_version`.

## Known Limits

- Pilotni enum `portal` je omezen na `firmy.cz` a `zivefirmy.cz`. Dalsi portaly vyzaduji schema version bump.
- `segment` je volny text s regexem, ne fixed catalog. Pokud A-03 narazi na potrebu fixed katalogu, je to breaking change.
- `max_results = 500` horni limit je odhad; realny limit urci A-04 / A-06 podle Serper quota a portal ToS.
- Zadna runtime validace v projektu jeste nebezi — pro runtime validaci pres AJV nebo rucni validator je potreba samostatny task (az A-02 nebo A-04).
