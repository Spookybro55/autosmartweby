## Scope

Connects the existing-but-dormant A-04 scraper pipeline (firmy.cz Node.js scraper +
A-02 _raw_import staging + A-03 normalizer + A-05 dedupe + A-10 batch orchestrator)
to a CRM frontend UI. Operator can dispatch scrape jobs from `/scrape` page, sees
duplicate-query alert before re-running the same search, and resolves cross-portal
dedupe candidates via side-by-side comparison in `/scrape/review`.

Cross-portal dedupe extension (Stream A primary value): existing dedupe matched on
IČO + website domain + email domain + name+city. New layer adds **phone exact match**
and **owned-domain email exact match** as primary signals — catches cases where the
same firm appears on different portals under slightly different names with
identical contact info.

Multi-portal extensibility designed in: portal name flows through types → trigger
payload → AS validation → GH workflow input → scraper CLI dispatch. Adding
`zivefirmy.cz` etc. requires only:
1. Append to `SUPPORTED_SCRAPE_PORTALS` (Config.gs + types/scrape.ts)
2. Add parser in `scripts/scraper/lib/{portal}-parser.mjs`
3. Add portal case in `scripts/scraper/{name}.mjs` dispatch (or unify into single `scraper.mjs`)

Auto-import: NEW_LEAD → LEADS, HARD_DUP → skip with log, SOFT/REVIEW → held in
`_raw_import` for operator review queue. Operator decides per-row: skip / merge
(fill-only-empty fields) / import-as-new (with explicit confirm).

## Known Limits / Decisions

- **Phone match always REVIEW**, never HARD_DUP. Same phone on two portals is strong signal but not conclusive — could be franchise or shared call center. Operator always decides via side-by-side dialog.
- **Free-domain email match (gmail/seznam) = REVIEW**, not HARD. Two leads with `info@gmail.com` are not the same person.
- **N vs 1 review case (raw row matches multiple existing leads via different signals — e.g. phone matches lead A, email matches lead B):** out of scope for A-11. Current implementation surfaces only the first-tier match. Backlog: dedupe extension to return multiple candidate matches and dialog UI for "which is the right one?".
- **Scraper portal dispatch:** currently hard-coded `case "firmy.cz"` in `scrape.yml`. Adding a 2nd portal in current state requires editing the workflow. Cleaner alternative for backlog: rename `scripts/scraper/firmy-cz.mjs` → `scraper.mjs` with `--portal` arg, single CLI handles all portals.
- **`processPreviewQueue` cron** runs every 15 min and may try to qualify scrape-imported leads before A-06 web check completes. Existing safeguards (lead_stage='NEW' default, processPreviewQueue only acts on 'qualified_for_preview=TRUE') prevent harm — newly imported leads sit at NEW until A-06 → autoQualifyHook lifts them.
- **Vercel `GITHUB_ACTIONS_TOKEN` env var** must be set manually in Vercel project Settings before scrape dispatch works. PAT requires `workflow:write` scope on `Spookybro55/autosmartweby`. Gracefully degraded behavior when unset (returns 200 + warning).
- **GH repo secrets** `APPS_SCRIPT_WEB_APP_URL` and `FRONTEND_API_SECRET` must be set in Settings → Secrets → Actions before workflow can ingest. Workflow fails with clear error if missing.
- **Cleanup of stray zero-byte files** (`0)`, `1)`, `5`, `maxVersion)`, `setupPreviewExtension``) — pre-existing pattern from shell-paste accidents in earlier sessions. Not touched by A-11.
