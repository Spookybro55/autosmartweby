UX decisions:
- Win rate denominator: `won / replied` (ne `won / sent`). Per spec — stricter, more meaningful pro sales konverzi (z odpovědí, kolik zavřených dealů). Sent → replied je separate funnel step.
- Summary box agreguje pouze `status === 'active'` templates. Archived versions ma kazdy svou kartu, ale neceleknuje se do rolling totals (jejich sent/replied/won ovlivňují historicke data, ne current performance).
- Card per VERSION (ne per key): pokud no-website ma v1 (archived) a v2 (active), uvidis 2 karty. Listing per key by zatemnoval performance srovnani mezi verzemi.
- Žádný cache — refresh button je explicit, fetch on mount. Backend AS-side `getTemplateAnalytics_` je single-pass LEADS scan (~50ms/800 rows), low cost.
- Zero-state na cards: kdyz `totals.sent === 0`, "Zatím žádné odeslané emaily s touto šablonou." (active template ale 0 sends — typicky just-published nebo bez fitting leadu).

## Known Limits / Decisions

- `generateTemplateId_` pattern `ASW-TPL-{ts_base36}-{rand4}` mirror `generateLeadId_`, zadny collision check (akceptovany risk pri low-volume <100 publikaci/den; collisions by se navic projevily pri T2 publish flow, ne v T1 bootstrap kde id generuje hned po sobe a `Math.random` rand4 prefix zarucuje rozliseni).
- `created_by` fallback `'system'` kdyz `Session.getActiveUser().getEmail()` vrati prazdny string (time-driven context kompatibilita; T1 bootstrap se vola z menu takze normalne se naplni emailem operatora).
- Load order komentar `4/5` v souborovem headeru je informacni — Apps Script alphabetizuje runtime load (E < H < M < P), takze realne `EmailTemplateStore.gs` se nacita mezi `Config.gs` a `Helpers.gs`. Komentar 4/5 vs 4/8 v `PreviewStore.gs` neresi konflikt — oba jsou jen popisne.
- T1 nepouzil `setupPreviewExtension` ze `setupEmailTemplates` jako transakcni jednotku — pokud `ensureEmailTemplatesSheet_` selze, LEADS uz pribyly. Akceptovano: oba kroky idempotentni a re-run je bezpecny.
- **T4 residual:** `DEFAULT_REPLY_TO_EMAIL = 'sebastian@autosmartweb.cz'` a `DEFAULT_REPLY_TO_NAME = 'Sebastián Fridrich'` zustavaji na pre-T4 legacy hodnotach, protoze spec explicitne rekla `resolveSenderIdentity_ NETKAT` a tyto konstanty cte prave ta funkce. Dusledek: pri Reply-To pro UNASSIGNED leady (`assignee_email = ''`) se pouzije starsi domena `sebastian@autosmartweb.cz` namisto noveho kanonickeho `s.fridrich@autosmartweb.cz`. V praxi nizke riziko (frontend obvykle prirazuje assignee pred sendem). Backlog: doreseni v naslednem PR ktery sjednoti `DEFAULT_REPLY_TO_*` s `DEFAULT_ASSIGNEE_PROFILE` a aktualizuje `resolveSenderIdentity_` aby cetla `getAssigneeProfile_` misto `ASSIGNEE_NAMES + DEFAULT_REPLY_TO_*`.
- **T4 idempotence pozor:** `migrateLegacyAssigneeEmails_` re-run je bezpecny (zadne legacy keys uz neexistuji po prvnim spusteni). Ale `bootstrapNoWebsiteV1` po prvnim publish zustane no-op — re-publish vyzaduje rucne pres frontend UI (T6) nebo `discardTemplateDraft_` + `saveTemplateDraft_` + `publishTemplate_` z editoru.
- **T9 manual override no-op:** Frontend posila `templateKey` v `regenerateDraft` body, ale AS T5 ho zatim honoruje pouze v doPost dispatch (`regenerateDraftForLead_` ho prijima ale ignoruje per docstring). Manualni template override ze drawer dropdown = no-op s budoucim T5.5 refactorem ktery rozsiri `composeDraft_` aby prijal explicit template injection. Akceptovany interim stav: operator vidi v dropdown ktera sablona byla pouzita, muze cist warningy, muze regenerovat s current chooseEmailTemplate_ logikou (auto-select pres resolveWebsiteState_).
- **T9 LeadListItem missing fields:** `/api/leads` list endpoint vraci `LeadListItem` ktery NEMA `area` / `segment` / `painPoint`. Pri preview proti realnemu leadu se tyto placeholders renderuji prazdne. Operator chcici fully-populated preview prepne na sample leadu (3 fixtures). Pro `segment` placeholder pouzivame `serviceType` jako best-effort approximaci (typicky stejna hodnota v praxi). Backlog: rozsirit `LeadListItem` o tyto fields nebo zvlast `/api/leads/preview-fixtures` endpoint.
