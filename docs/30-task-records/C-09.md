## Scope

Formalizuje **centralizovanou exception queue** a **operator resolution kontrakt** pro pripady, ktere automat neumi / nesmi rozhodnout sam. Misto "tichych" skipu a sheet error sloupcu roztrousenych po LEADS (`preview_error`, `email_reply_type=UNCLASSIFIED`, `bounce_class=SOFT`) nebo `_asw_dead_letters` (CS3 technicky retry exhaustion) definuje C-09 **jeden human-facing sheet** (`_asw_exceptions`) s jasnym resolution kontraktem, priority modelem, SLA targety, auditabilitou a deterministickym flow re-entry.

Scope je **SPEC-only** — neimplementuje runtime review worker, cron trigger, queue worker, mailbox sync runtime, ESP webhook, frontend UI (B6), suppression list management, AI-based auto-triage ani zapisy do `apps-script/Config.gs`. Vsechny nove artefakty jsou oznacene **PROPOSED FOR C-09** a budou materializovany implementacnim taskem. B6 (operator exception dashboard UI) **NENI blocker** pro C-09 SPEC — operator muze v interim resolvovat exceptions primo v Google Sheets bunce (`_asw_exceptions` row edit).

## Known Limits

- SPEC neimplementuje engine runtime / cron / scheduler / SLA auto-bump cron / `detectException()` hooks — pouze definuje kontrakt. Implementace je budouci task (`detectException()` helper + `resolveException()` dispatcher + sheet creation + Config.gs + `_asw_logs` event emission).
- PROPOSED `_asw_exceptions` sheet neni v ramci C-09 vytvoren. Implementacni task ho musi pridat spolu s materializaci detection engine.
- PROPOSED enumy (`exception_type`, `exception_priority`, `exception_status`, `operator_decision`, `resolution_outcome`, `next_action`) nejsou v ramci C-09 zapsany do `apps-script/Config.gs`. Implementacni task je musi pridat.
- PROPOSED C-04 `MANUAL_REVIEW_OVERRIDE` context parameter je extension, do materializace pouziva operator interim pres C-04 fallback logic (degraded).
- PROPOSED C-05 idempotency_key extension (`retry_of=exception_id` suffix) je extension; do materializace retry queue rows musi byt rucne trackovane pres `retry_reference_queue_id` na exception.
- B6 frontend UI je **mimo C-09 scope**. V interim operator resolvuje primo v Google Sheets bunce (`_asw_exceptions` row edit + on-edit trigger). Degraded UX (bez per-lead aggregate view, bez notification).
- Suppression list centralization je **mimo C-09 scope** (budouci C-10 task). C-09 poskytuje source data (compliance_hard_stop + unsubscribed audit trail).
- Notification system (email alerts, Slack integration pro new P1 exception) je **mimo C-09 scope**. Operator musi aktivne kontrolovat `_asw_exceptions` sheet.
- Exception archive / retention policy je **dokumentovana jen principe** (append-only, no delete). Detail policy = ops rozhodnuti + budouci task.
- AI-based auto-triage / priority prediction je **mimo v1.0 SPEC** (v2.0 mozna add priority re-sort algoritmy).
- Per-operator SLA / workload balancing je **mimo SPEC** (ops concern).
- Multi-tenant exception routing je **mimo v1.0** (single-tenant).
- SLA auto-bump rules jsou **PROPOSED** — konkretni thresholds (2 SLA period pending, 14 dni stale, 7 dni OOO hold) jsou doporucene defaults. Ops rozhodne finalni hodnoty.
- Retry chain depth limit=3 je **PROPOSED default**. Pokud implementator chce hlubsi chain, muze zvetsit (ale exception_retry_chain_broken alert je treba recalibrate).
- Recent-closed window=7 dni je **PROPOSED default**. Kratsi window = vice duplicitnich exceptions pri rychlem re-failu; delsi = operator muze zmeskat opakovany problem.
- Compatibility matrix validation je **dokumentovana v SPEC** (napr. `compliance_hard_stop` + `approve` = validation error) ale **neni runtime enforced** — implementacni task musi pridat validation logic.
- `detectException()` hooks v A-02/A-03/A-05/A-06/A-07/A-08/B-04/C-04/C-05/C-06/C-07/C-08 jsou **PROPOSED integration points**. Kazdy hook musi byt pridan implementacnim taskem bez regression existing step behavior.
- Dedup_key patterns jsou **PROPOSED** — specifickeho kolisi potencial (napr. co kdyz stejny lead ma 2 ambiguous_duplicate proti 2 ruznym candidates? Resi unordered composite key `primary_lead_id:candidate_lead_id` sorted alphabetically). Implementacni task musi pridat unit tests.
- `CLOSED_STALE` auto-transition po 14 dnech je **audit-preserving** (ne reject) — znamena "operator nerozhodnul, system zavrel". Analytics metrics musi to rozlisovat od aktivne `rejected`.
- `exception_retry_chain_broken` alert pri depth=3 je **dokumentovana jako event type** ale **neni notification mechanism defined** (kam se posila, kdo resi). Notification je mimo SPEC.
- `operator_edited_fields_json` max velikost 4KB je **PROPOSED limit** align s `diagnostic_payload_json`. Pokud operator chce editovat vic poli, musi pridat pokracovaci exception.
- Audit immutability po `CLOSED` je **dokumentovana jako invariant** ale **Google Sheets nema native immutability** — implementacni task musi pridat edit-protection (napr. data validation rule + protected range pro CLOSED/CANCELLED rows).
