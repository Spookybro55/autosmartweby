- D10. **Visual check result** — klasifikovat jako `OK`, `partial-ok` (drobne issues), `broken`.

### E. Pass/fail classification

**PASS pokud vsechno plati:**
- Ending `preview_stage ∈ {READY_FOR_REVIEW, APPROVED}` (APPROVED nesmi prijit automaticky behem execution, ale ma byt tolerovano pokud ho operator legitimne vynutil mimo pipeline — nedoporucuje se behem measurement window).
- `preview_url` vyplneny.
- URL vraci HTTP 200 a zobrazuje stranku.
- Visual check je `OK` nebo `partial-ok`.

**FAIL pokud alespon jeden plati:**
- Ending `preview_stage = FAILED`.
- `preview_url` prazdny (ale stage nebyl `FAILED`).
- URL 4xx/5xx nebo network error.
- Visual check je `broken`.
- Critical data mismatch (napr. hero zobrazuje cizi firmu).

Po dokonceni 10 leadu:

- F1. Spocitat `pass_count`, `fail_count` — ocekavany threshold `pass_count >= 7`.
- F2. Vypsat fail reasons (unique + frequency) — napr. "webhook 401 auth", "unknown template_type fallback broken", "timeout".
- F3. Vybrat **3–5 ukazkovych preview URL** (preferovat mix: 2-3 OK kvalitni + 1-2 s `preview_needs_review=true`) a zaznamenat je do summary sekce.
- F4. Vypracovat doporuceni:
  - Pokud `pass_count >= 7` — pilot PASSED, B-08 lze oznacit `DONE` v samostatnem execution-evidence commitu.
  - Pokud `pass_count < 7` — pilot FAILED, B-08 zustava `READY_FOR_EXECUTION` a identifikovane issues se rozpadnou do follow-up tasku (navrh IDs: B-08-fix-1..N nebo samostatne A/B/C tasky podle vrstvy).
