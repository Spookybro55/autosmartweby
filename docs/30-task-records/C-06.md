## Scope

Formalizuje provider-agnostickou vrstvu mezi `_asw_outbound_queue` a konkretnim ESP (Gmail / SendGrid / Mailgun / …). Oddeluje ctyri vrstvy identity statusu: provider raw response / normalized provider status / queue send status / CS1 lifecycle state. Definuje `EmailSender` interface (1 metoda), `SendRequest` (17 poli), `NormalizedSendResponse` (17 poli), `NormalizedProviderStatus` (7 hodnot), `NormalizedSendErrorClass` (8 hodnot) s fixnim mappingem na CS3 `failure_class`, rate limiting jako kontrakt, 3 fail scenare, Gmail vs generic ESP sample mapping a sender selection via config (ne runtime).

Scope je **SPEC-only** — zadny runtime sender, Gmail adapter, SendGrid adapter, Mailgun adapter, queue worker, factory, mailbox sync, frontend UI ani provider webhook ingest se neimplementuje. Zadne nove enumy ani Script Property se v tomto tasku nezapisuji do `apps-script/Config.gs`. Vsechny nove artefakty jsou oznacene PROPOSED FOR C-06 a budou materializovany implementacnim taskem.

**CS3 kompatibilita:**
- `NormalizedSendErrorClass` (8 hodnot) je jemnejsi nez CS3 `failure_class` (TRANSIENT / PERMANENT / AMBIGUOUS). C-06 definuje **deterministicky 1:N lookup table** (TIMEOUT→AMBIGUOUS, RATE_LIMIT→TRANSIENT, INVALID_RECIPIENT→PERMANENT, AUTH_FAILED→PERMANENT, PROVIDER_UNAVAILABLE→TRANSIENT, PROVIDER_REJECTED→PERMANENT, INVALID_REQUEST→PERMANENT, UNKNOWN→AMBIGUOUS). Queue worker mapuje 1:N.
- CS3 S12 `process_email_queue` invariant `max_attempts=1` je respektovany — `EmailSender.send` nikdy nerestrykuje interne. Retry = novy queue row (C-05 pravidlo). C-06 pouze emituje `retryable: boolean` hint pro diagnostiku.
- C-06 neemituje `_asw_logs` sam; queue worker to dela s normalized payloadem. `sender_run_id` + `sender_event_id` jsou echoed zpet v response.

**C-05 kompatibilita:**
- `SendRequest` je stavebni derivat C-05 payload kontraktu v1.0 (sekce 6 docs/24). 13 poli je immutable snapshot z queue row; 4 pole (`sender_run_id`, `sender_event_id`, `timeout_ms`, `payload_version`) jsou runtime-derived.
- `NormalizedSendResponse.provider_message_id` + `sent_at` jsou pole, ktera queue worker zapise do queue row pri `QUEUED → SENT` transition.
- Queue statusy (`QUEUED`, `SENDING`, `SENT`, `FAILED`, `CANCELLED`) nejsou touto SPEC dotcene — ortogonalne existuji vuci `NormalizedProviderStatus`.
- C-05 `idempotency_key` je predavan do `SendRequest.idempotency_key`; adapter rozhoduje, zda ho propne do provider API (SendGrid `X-Message-Id` header) nebo drzi pouze lokalne (Gmail nema native idempotency).

**CS1 kompatibilita:**
- C-06 nezavadi zadny novy canonical lifecycle state. T18 (`EMAIL_QUEUED → EMAIL_SENT`) je triggered queue workerem po `success: true`. C-06 samotny nezapisuje do LEADS.
- C-06 fail → CS1 zustava `EMAIL_QUEUED`. Transition na potencialny `EMAIL_FAILED` je manualni operator akce (T25), ne C-06 responsibility.

**C-04 kompatibilita:**
- C-04 zije **pred** queue; C-06 zije **za** queue. Zadna prima interakce. Oddelene C-05 queue vrstvou.

## Known Limits

- SPEC neimplementuje adaptery ani queue worker — pouze definuje kontrakt mezi nimi. Implementace je budouci task.
- PROPOSED enumy nejsou v ramci C-06 pridany do `apps-script/Config.gs`. Implementacni task je musi pridat spolu s materializaci C-05 PROPOSED poli.
- C-06 definuje pouze **success** CS1 transition (`EMAIL_QUEUED → EMAIL_SENT`). `EMAIL_FAILED` CS1 state neexistuje v CS1 18-state machinu; fail je handled na urovni queue worker `FAILED` + `failure_class` + dead-letter. Pokud by se zavedl `EMAIL_FAILED` canonical state, je to CS1 change, ne C-06.
- Rate limiting je C-06 pouze contract (adapter report `rate_limit_reset_at` + `THROTTLED`). Scheduling, quiet hours, daily caps jsou C-08 — C-06 je agnosticky.
- Multi-provider fallback (primary Gmail + secondary SendGrid) je explicitne out-of-scope. Sender selection je single-active-provider per deployment. Pokud deployment vyzaduje rostouci sophistication, je to budouci task.
- Gmail nema native API idempotency. Adapter reuse `outreach_stage` monotonic guard + 5min double-send window (existing `OutboundEmail.gs` pattern) jako pseudo-idempotency. ESP adaptery mohou propnout `idempotency_key` do provider headeru (SendGrid `X-Message-Id`).
- Attachment support je v1.0 rezerva — `SendRequest.attachments` je vzdy prazdne pole. Implementacni task pro attachments je budouci extension.
- HTML body je v1.0 rezerva — `SendRequest.body.html` je optional. Queue worker zatim predava pouze `body.plain`.
- Thread reply (follow-up) je v1.0 rezerva — `SendRequest.thread_hint` je pole pro C-07. Gmail adapter ho propne do reply-in-thread, ESP adapter do `In-Reply-To` headeru.
- Provider webhook ingest (bounce, complaint, open, click) neni soucasti C-06. Cross-ref na `provider_message_id` je pripraveny, implementace je budouci task po C-07/C-08.
- `provider_response_excerpt` ma 256-char limit pro storage efficiency v `_asw_logs`. Plny provider response se neulozi; implementacni task muze rozhodnout o separatnim `_asw_provider_raw` append-only sheetu pokud to CS3 audit bude vyzadovat.
- PII safety invariant se tyka pouze logged poli (`provider_response_excerpt`, `error_message`). Send payload (subject + body) je PII-containing, ale to je C-05 queue vec, ne C-06.
- C-06 neresi rozdil mezi soft bounce a hard bounce — provider webhook ingest je mimo scope. `INVALID_RECIPIENT` error_class pokryva bounce synchronne (pokud ESP ho reportuje v real-time response), ale asynchronni bounce signal je C-09 territory.
